\# 🧨 จุดที่ “ควรแก้” (มีผลจริง)

\#\# 1\) \`parseFloat(...) || 0\` อาจกลบค่าผิดพลาด

\`\`\`js  
const rawLat \= parseFloat(row\[SRC\_IDX.LAT\]) || 0;  
const rawLng \= parseFloat(row\[SRC\_IDX.LNG\]) || 0;  
\`\`\`

👉 ปัญหา:

\* ถ้า parse ไม่ได้ → \`NaN\`  
\* แต่ \`NaN || 0\` → กลายเป็น \`0\`

\#\#\# 💥 ผล:

\* แยกไม่ออกว่า “0 จริง” หรือ “parse พัง”

\#\#\# ✅ แนะนำ:

\`\`\`js  
const rawLat \= Number(row\[SRC\_IDX.LAT\]);  
const rawLng \= Number(row\[SRC\_IDX.LNG\]);

const hasGeo \= \!isNaN(rawLat) && \!isNaN(rawLng) && rawLat \!== 0 && rawLng \!== 0;  
\`\`\`

\---

\#\# 2\) \`hasGeo\` logic ยังไม่กัน NaN

\`\`\`js  
const hasGeo \= rawLat \!== 0 && rawLng \!== 0;  
\`\`\`

👉 ถ้า rawLat \= NaN  
→ \`NaN \!== 0\` \= true ❗

\#\#\# 💥 ผล:

\* จะถือว่ามีพิกัด ทั้งที่จริง “พัง”

\#\#\# ✅ แก้:

\`\`\`js  
const hasGeo \= \!isNaN(rawLat) && \!isNaN(rawLng) && rawLat \!== 0 && rawLng \!== 0;  
\`\`\`

\---

\#\# 3\) Cache อาจแตก (size limit)

\`\`\`js  
cache.put(cacheKey, JSON.stringify(result), AI\_CONFIG.CACHE\_TTL\_SEC);  
\`\`\`

👉 Apps Script:

\* limit \~100KB ต่อ key

\#\#\# 💥 ถ้า data เยอะ:

\* cache.put จะ throw silently (คุณ catch ไว้แล้ว ✔️)

👉 แต่:

\* ไม่มี log → debug ยาก

\#\#\# ✅ เพิ่ม:

\`\`\`js  
} catch (e) {  
  logWarn('SourceRepo', 'Cache ใหญ่เกิน — ไม่ได้ cache');  
}  
\`\`\`

\---

\#\# 4\) \`processSrcBatch\_\` กลืน error ทั้ง batch 😶

\`\`\`js  
try {  
  batch.forEach(srcObj \=\> processOneRow(srcObj));  
} catch (err) {  
\`\`\`

👉 ถ้า row เดียวพัง:

\* batch ทั้งก้อนหยุด

\#\#\# 💥 ผล:

\* data หายเงียบ

\#\#\# ✅ แนะนำ:

\`\`\`js  
batch.forEach(srcObj \=\> {  
  try {  
    processOneRow(srcObj);  
  } catch (err) {  
    logError('SourceRepo', \`Row fail: ${srcObj.invoiceNo} — ${err.message}\`);  
  }  
});  
\`\`\`

\#\#\# ✅ ทางเลือก:

# **perplexity**

ไฟล์ \`04\_SourceRepository.gs\` ของคุณเขียนออกมาได้ดีเยี่ยมในแง่ของโครงสร้างและการจัดการ Memory ครับ ระบบอ่านข้อมูลแบบ Batch และการทำ Cache ข้อมูลต้นทางถือว่าทำได้ตามมาตรฐาน \*\*LMDS V5.0\*\* ที่คุณวางไว้

จากการตรวจสอบโค้ดอย่างละเอียด มีจุดที่ควรระวังและแก้ไขเพื่อให้ระบบทำงานได้แม่นยำขึ้น ดังนี้ครับ:

\#\#\# จุดที่พบ (ต้องแก้ไข)

| จุดที่พบ | ปัญหา | วิธีแก้ไข |  
| :--- | :--- | :--- |  
| \*\*การเข้าถึง Object\*\* | ใน \`getUnprocessedRows\` คุณเขียน \`\!doneSet.has(row.invoiceNo)\` แต่ใน \`buildSourceObj\_\` คุณใช้ property ชื่อ \`invoiceNo\` (ถูกต้อง) \*\*แต่\*\* ในบางส่วนของระบบหากมีการเรียกใช้ผ่าน \`FACT\_IDX\` ต้องมั่นใจว่าใช้ชื่อ Key เดียวกันตลอด | ตรวจสอบให้แน่ใจว่าโปรเจกต์คุณใช้ \`invoiceNo\` เป็นมาตรฐาน (CamelCase) ในทุก Object |  
| \*\*ความผิดพลาดใน \`processSrcBatch\_\`\*\* | โค้ดเรียก \`batch.forEach(srcObj \=\> processOneRow(srcObj))\` หากฟังก์ชัน \`processOneRow\` ยังไม่ได้เขียนไว้ในไฟล์นี้ หรือในไฟล์อื่นๆ ที่เชื่อมต่อกัน โค้ดจะ error ทันที | ตรวจสอบว่า \`processOneRow\` ถูกประกาศไว้ใน \`10\_MatchEngine.gs\` และมีการ \`import\` หรืออยู่ในโปรเจกต์เดียวกัน |  
| \*\*การ Cache ข้อมูลขนาดใหญ่\*\* | \`CacheService.getScriptCache()\` มีข้อจำกัด (Limit) ที่ \*\*100KB ต่อรายการ\*\* หากข้อมูลในชีต \`SCGนครหลวงJWDภูมิภาค\` มีจำนวนแถวมากเกินไป การ \`JSON.stringify(result)\` จะเกินขนาดและทำให้ Cache ล้มเหลว | ควรใส่ \`try-catch\` ครอบการ \`put\` ลง Cache (ซึ่งคุณทำแล้ว ถือว่าดีมาก) แต่ถ้า Cache ล้มเหลวบ่อยๆ แนะนำให้เพิ่ม Log เพื่อแจ้งเตือน |

\#\#\# ข้อแนะนำเพิ่มเติม (Pro Tips)  
1\. \*\*การตรวจสอบ \`rowNum\`:\*\* ใน \`runLoadSource\` คุณคำนวณ \`i \+ 2\` ซึ่งถูกต้องสำหรับชีตที่มี Header อยู่แถว 1 แต่หากในอนาคตชีตมีการ Insert แถวว่างข้างบน Header อาจทำให้เลข Row เพี้ยนได้ (เพียงแค่ต้องระวังไว้ครับ)  
2\. \*\*การทำ Validation ข้อมูล:\*\* ใน \`buildSourceObj\_\` คุณมีการทำ \`parseFloat\` ซึ่งดีมากครับ แต่แนะนำให้เพิ่ม \`if (isNaN(rawLat)) rawLat \= 0;\` เพื่อป้องกันค่าที่อาจจะติดมาเป็น \`NaN\` ซึ่งจะทำให้การคำนวณ Distance ใน Match Engine พังได้  
3\. \*\*การจัดการ Set:\*\* ใน \`getProcessedInvoiceSet\_\` คุณใช้ \`String(r\[0\]).trim()\` ซึ่งถูกต้องและรัดกุมมากครับ เพื่อป้องกันปัญหาเรื่อง Space หรือ Data Type ที่ต่างกัน

\#\#\# สรุปภาพรวม  
โค้ดไฟล์นี้ผ่านเกณฑ์มาตรฐานครับ โครงสร้างมีความต่อเนื่องจากไฟล์ก่อนหน้าได้อย่างไร้รอยต่อ โดยเฉพาะการดึงข้อมูล \`FACT\_DELIVERY\` มาเป็น \`Set\` เพื่อทำ \`Filter\` ใน \`getUnprocessedRows\` นั้นเป็นเทคนิคที่ทำให้โค้ดทำงานได้เร็วมาก (Big O Notation ดีเยี่ยม)

\*\*สามารถนำโค้ดนี้ไปรันได้เลยครับ\*\* หากมั่นใจว่าฟังก์ชัน \`processOneRow\` ใน \`10\_MatchEngine.gs\` พร้อมใช้งานแล้ว\!

# **qwen**

จากการตรวจสอบโค้ด \`04\_SourceRepository.gs\` อย่างละเอียด \*\*พบจุดที่เขียนผิดทางเทคนิค (Syntax/Typo) 1 จุด\*\* และ \*\*จุดที่ควรตรวจสอบความสอดคล้อง 1 จุด\*\* ดังนี้ครับ:

\#\#\# 🚨 1\. จุดที่เขียนผิด (Critical Typo)  
ในฟังก์ชัน \`buildSourceObj\_()\` มีการเคาะวรรคผิดในตัวแปร \`rawLat\` ทำให้โค้ดอาจจะพังหรืออ่านค่าผิดครับ

\*\*ตำแหน่งที่ผิด:\*\*  
\`\`\`javascript  
// บรรทัดที่ \~195  
    rawAddress:    String(row\[SRC\_IDX.SYS\_ADDRESS\] || '').trim(),  
    rawLat:        rawLat,  // ❌ ผิด: มีช่องว่างในชื่อ key (rawLat: )  
    rawLng:        rawLng,  
\`\`\`

\*\*วิธีแก้ไข:\*\*  
ลบช่องว่างระหว่าง \`rawLat\` กับ \`:\` ออกครับ  
\`\`\`javascript  
    rawAddress:    String(row\[SRC\_IDX.SYS\_ADDRESS\] || '').trim(),  
    rawLat:        rawLat,  // ✅ ถูก: ไม่มีช่องว่าง  
    rawLng:        rawLng,  
\`\`\`  
\> \*\*เหตุผล:\*\* แม้ JavaScript บางครั้งจะยืดหยุ่น แต่การเขียน Object Key แบบนี้เสี่ยงต่อ Syntax Error และการเรียกใช้ \`srcObj.rawLat  
at\` ในไฟล์อื่นจะทำได้ยากครับ

\---

\#\#\# ⚠️ 2\. จุดที่ควรตรวจสอบตรรกะ (Logic Check)  
ในฟังก์ชัน \`getProcessedInvoiceSet\_()\` มีการคำนวณจำนวนแถว (\`numRows\`) ในการดึงข้อมูล:

\`\`\`javascript  
// บรรทัดที่ \~137  
const lastRow      \= factSheet.getLastRow() \- 1;   
// ...  
const invoiceData  \= factSheet.getRange(2, invoiceCol, lastRow, 1).getValues();  
\`\`\`

\*\*ข้อสังเกต:\*\*  
\*   \`getRange(row, column, numRows, numColumns)\` พารามิเตอร์ที่ 3 คือ \*\*จำนวนแถว\*\* ที่จะดึง  
\*   ถ้า \`factSheet.getLastRow()\` คือ 10 (มีข้อมูลแถวที่ 2-10 รวม 9 แถว)  
\*   โค้ดคำนวณ \`lastRow \= 10 \- 1 \= 9\` → ดึง 9 แถว → \*\*ถูกต้องครับ\*\* ✅

แต่หากชีตว่างเปล่า (\`getLastRow()\` returns 1):  
\*   \`lastRow \= 1 \- 1 \= 0\` → ดึง 0 แถว → \*\*ปลอดภัยครับ\*\* ✅

\*\*สรุป:\*\* ตรรกะนี้เขียนถูกต้องแล้วครับ เพียงแต่อยากให้เห็นภาพชัดเจนว่าทำไมต้อง \`-1\` (เพราะเริ่มดึงจากแถวที่ 2 ไม่ใช่แถวที่ 1\)

\---

\#\#\# ✅ 3\. สิ่งที่เขียนถูกต้องแล้ว (สังเกตว่าพัฒนาขึ้น)  
\*   \*\*การขึ้นบรรทัดใหม่:\*\* ใช้ \`\\n\` ถูกต้องทั้งหมด (ไม่มีการพิมพ์ผิดเป็น \`\\\\n\` เหมือนไฟล์แรก)  
\*   \*\*การจัดการ Cache:\*\* การใช้ \`CacheService\` ใน \`getAllSourceRows()\` มี \`try-catch\` ครอบไว้ดีแล้ว เพื่อป้องกันกรณีข้อมูลใหญ่เกินหรือเกิดข้อผิดพลาด  
\*   \*\*การวนลูป:\*\* การใช้ \`filter\` และ \`map\` ใน \`getAllSourceRows()\` เขียนได้กระชับและถูกต้องตามหลัก JavaScript Modern

\---

\#\#\# 💡 4\. ข้อแนะนำเพิ่มเติม (Best Practices)

\*\*4.1 การจัดการ \`processOneRow\`\*\*  
ในฟังก์ชัน \`processSrcBatch\_()\`:  
\`\`\`javascript  
batch.forEach(srcObj \=\> processOneRow(srcObj));  
\`\`\`  
\*   \*\*ข้อสังเกต:\*\* ฟังก์ชันนี้เรียก \`processOneRow\` ซึ่งน่าจะเป็นฟังก์ชันหลักในไฟล์ \`10\_MatchEngine.gs\`  
\*   \*\*คำแนะนำ:\*\* เนื่องจาก GAS รันโค้ดแบบเรียงบรรทัด (Synchronous) การเรียกแบบนี้ในลูปอาจจะช้าถ้า \`processOneRow\` มีการเรียก API ภายนอก ควรตรวจสอบว่าในไฟล์ \`10\_MatchEngine.gs\` มีการจัดการ \`Utilities.sleep()\` หรือการหน่วงเวลาเพื่อไม่ให้ติด Quota ของ Google ครับ

\*\*4.2 ความสม่ำเสมอของชื่อตัวแปร\*\*  
ใน \`buildSourceObj\_()\` มีการส่งค่าออก:  
\`\`\`javascript  
    rawPersonName: String(row\[SRC\_IDX.SHIP\_TO\_NAME\] || '').trim(),  
    rawAddress:    String(row\[SRC\_IDX.SYS\_ADDRESS\] || '').trim(),  
\`\`\`  
\*   \*\*ข้อสังเกต:\*\* ใช้ชื่อ \`rawPersonName\` และ \`rawAddress\`  
\*   \*\*คำแนะนำ:\*\* ตรวจสอบในไฟล์ \`02\_Schema.gs\` (ชีต \`FACT\_DELIVERY\` หรือ \`Q\_REVIEW\`) ว่าชื่อคอลัมน์ปลายทางสอดคล้องกันหรือไม่ (เช่น \`raw\_person\_name\` vs \`rawPersonName\`) เพื่อให้การจับคู่ข้อมูล (Mapping) ทำได้ถูกต้องครับ

\`\`\`javascript

\`\`\`javascript

      processOneRow(srcObj);

      logError('SourceRepo', \`แถว ${srcObj.sourceRow} ล้มเหลว: ${err.message}\`);

  });  
  return failed;  
}

โค้ดนี้ถูกเขียนได้สมบูรณ์แบบมาก ทั้งการจัดการข้อผิดพลาด การคำนึงถึงหน่วยความจำ  
(Memory Management) และความเร็วในการประมวลผล (Performance Optimization)

เพื่อให้มั่นใจ นี่คือจุดยอดเยี่ยมที่คุณทำถูกทั้งหมด  
และอยากให้รักษามาตรฐานนี้ไว้ครับ:

1\.  ปลอดภัยจากปัญหา \\n: คราวนี้การสร้าง Alert หรือ throw new Error() ทุกจุดใช้  
    Backtick () ครอบ หรือบรรทัดเดี่ยวได้ถูกต้องแล้วครับ  
    ไม่ตกม้าตายเรื่องแบ็กสแลชเบิ้ล  
    \\\` เหมือนช่วงก่อนแล้ว

2\.  ประสิทธิภาพสูง (Memory Caching & Row Reading): การเลือกใช้ getValues()  
    ดึงก้อนข้อมูลทั้งตารางออกมาเก็บไว้ใน allRows เพียงครั้งเดียว  
    เป็นวิธีการดึงข้อมูลจาก Spreadsheet ที่รวดเร็วที่สุด ดีมากครับ

3\.  ป้องกันขนาดไฟล์บวมตอนสร้าง Cache (try...catch บน CacheService):  
    คุณเขียนรองรับไว้ได้อย่างหลักแหลมในบรรทัด:

    try {  
      cache.put(cacheKey, JSON.stringify(result), AI\_CONFIG.CACHE\_TTL\_SEC);  
    } catch (e) {  
      // ข้อมูลใหญ่เกินกว่าจะ Cache — ข้ามไป  
    }

    เนื่องจากข้อจำกัดของ CacheService จะเก็บสตริงได้ขนาดสูงสุดไม่เกิน 100KB  
    ต่อ 1 คีย์ หากข้อมูลการส่งของใน Source เยอะจนเกิน 100KB  
    โค้ดตรงส่วนนี้จะไม่พังพินาศ  
    แต่ระบบจะประณีประนอมแล้วคำนวณสดเอา  
    (ตามที่ใส่บล็อก try..catch ดักเอาไว้เป๊ะ) อันนี้โปรเท่มากครับ

4\.  Batch Processing ลดอาการ Timeout: การสร้าง if (batchRows.length \>=  
    AI\_CONFIG.BATCH\_SIZE \* 5\) ในจังหวะยิง API ส่งต่อ  
    เป็นทางเลือกที่ดีเยี่ยมในการกัน  
    Script ใช้งาน Time limits เกินกำหนด
