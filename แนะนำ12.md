\#\# ❌ 1\. resolvePlace ใช้พารามิเตอร์ผิด (ซ้ำจุดเดิมอีกแล้ว)

\`\`\`js  
const placeResult  \= resolvePlace(rawPlaceName, rawPlaceName);  
\`\`\`

👉 function จริง:

\`\`\`js  
resolvePlace(rawName, rawAddress)  
\`\`\`

⚠️ ตอนนี้:

\* rawName \= ✔  
\* rawAddress \= ❌ (เอาชื่อไปแทนที่อยู่)

\#\#\# ✔ แก้:

\`\`\`js  
const placeResult \= resolvePlace(rawPlaceName, '');  
\`\`\`

หรือถ้ามี address จริง:

\`\`\`js  
const placeResult \= resolvePlace(rawPlaceName, rowData\[REVIEW\_IDX.RAW\_SYS\_ADDR\]);  
\`\`\`

\#\# ❌ 2\. status column ใช้ชื่อ header ผิด case

\`\`\`js  
const statusCol     \= headers.indexOf('status')      \+ 1;  
const reviewedAtCol \= headers.indexOf('reviewed\_at') \+ 1;  
\`\`\`

แต่ตอน enqueue:

\`\`\`js  
newRow\[REVIEW\_IDX.STATUS\]      \= 'Pending';  
newRow\[REVIEW\_IDX.REVIEWED\_AT\] \= '';  
\`\`\`

👉 ชัดเลยว่า header \= \`STATUS\`, \`REVIEWED\_AT\` (uppercase)

⚠️ ปัญหา:

\* indexOf('status') → \-1  
\* \+1 \= 0  
\* ไปเขียน column 0 ❌ → error เงียบ / เขียนผิดช่อง

\#\#\# ✔ แก้:

\`\`\`js  
headers.indexOf('STATUS')  
headers.indexOf('REVIEWED\_AT')  
\`\`\`

หรือโคตรชัวร์:

\`\`\`js  
const statusCol     \= REVIEW\_IDX.STATUS \+ 1;  
const reviewedAtCol \= REVIEW\_IDX.REVIEWED\_AT \+ 1;  
\`\`\`

\#\# ❌ 4\. hasGeo logic พัง (ซ้ำ pattern เดิม)

\`\`\`js  
hasGeo: \!\!(Number(rowData\[REVIEW\_IDX.RAW\_LAT\])),  
\`\`\`

👉 ปัญหา:

\* lat \= 13.7 → true ✔  
\* lat \= 0 → false ✔  
\* lat \= NaN → false ✔

⚠️ แต่:

\* ไม่เช็ค lng  
\* lat มีค่าแต่ lng ไม่มี \= ยังถือว่า hasGeo \= true ❌

\#\#\# ✔ แก้:

\`\`\`js  
hasGeo: \!\!(Number(lat) && Number(lng))  
\`\`\`

หรือ:

\`\`\`js  
hasGeo: \!isNaN(lat) && \!isNaN(lng) && lat \!== 0 && lng \!== 0  
\`\`\`

# **perplexity**

ไฟล์ \`12\_ReviewService.gs\` ถือเป็นส่วนที่ปิดท้ายกระบวนการ Matching ได้อย่างสมบูรณ์ครับ การเพิ่ม \`Place Alias Learning\` เข้าไปใน \`MERGE\_TO\_CANDIDATE\` คือจุดสำคัญที่จะทำให้ระบบ "ฉลาดขึ้นเรื่อยๆ" ตามการใช้งานจริงของผู้ตรวจสอบ

จากการตรวจสอบโค้ด \*\*ไม่พบจุดที่ผิดพลาดรุนแรง (Critical Bug)\*\* ครับ ทุกอย่างเขียนมาได้ตรงตาม Schema และ Logic ที่วางไว้ แต่อยากให้สังเกต 3 ประเด็นเพื่อความแม่นยำในการใช้งานครับ:

\#\#\# ข้อแนะนำเชิงเทคนิค (Data Reliability)

| จุดที่พบ | ปัญหา | วิธีแก้ไข / ข้อเสนอแนะ |  
| :--- | :--- | :--- |  
| \*\*Logic การเช็ก status ใน Loop\*\* | ใน \`applyAllPendingDecisions\` คุณเช็กว่า \`status \=== 'In\_Review'\` แต่ใน \`enqueueReview\` คุณตั้งค่าเริ่มต้นไว้ที่ \`Pending\` | \*\*ต้องระวัง:\*\* หาก Reviewer ยังไม่ได้เปลี่ยนสถานะจาก \`Pending\` เป็น \`In\_Review\` ในชีต ระบบนี้จะข้ามแถวเหล่านั้นไปทั้งหมด แนะนำให้เช็กทั้ง \`Pending\` และ \`In\_Review\` หรือให้ระบบเปลี่ยนเป็น \`In\_Review\` อัตโนมัติเมื่อกดเริ่มงาน |  
| \*\*การใช้ \`getValues()\[0\]\`\*\* | ใน \`applyReviewDecision\` คุณอ่านค่า \`headers\` และ \`rowData\` โดยใช้ \`getValues()\[0\]\` ซึ่งเป็นวิธีที่เร็ว แต่ต้องระวังกรณีที่ชีตถูกแทรกแถวหรือโครงสร้าง Column เปลี่ยนไป | ระบบคุณวาง Schema ไว้ดีมากแล้ว วิธีนี้จึงปลอดภัยครับ แต่อยากเตือนให้รักษาลำดับ Column ในไฟล์ \`02\_Schema\` ให้คงที่เสมอ |  
| \*\*การสร้าง GeoPoint ในเคส CREATE\_NEW\*\* | ในกรณี \`CREATE\_NEW\` คุณสร้าง \`srcObj\` ขึ้นมาใหม่และเรียก \`executeDecision\` ซ้ำ | โครงสร้างนี้ดีครับ แต่ต้องมั่นใจว่าทุกครั้งที่สร้างใหม่ ระบบจะไม่วน Loop ซ้ำซ้อน (ดูจาก Logic แล้วไม่มีปัญหาครับ) |

\#\#\# ข้อแนะนำเพิ่มเติมเพื่อความแข็งแกร่ง  
1\. \*\*การปรับ Status:\*\* ใน \`applyReviewDecision\` เมื่อทำเสร็จคุณเปลี่ยนสถานะเป็น \`'Done'\` ซึ่งดีมาก แต่สำหรับกรณี \`ESCALATE\` คุณยังไม่มีโค้ดจัดการสถานะ แนะนำให้เพิ่ม \`sheet.getRange(reviewRow, statusCol).setValue('Escalated')\` เข้าไปด้วยครับ  
2\. \*\*Performance ของ \`highlightHighPriorityReviews\`:\*\* การใช้ \`setBackground\` ใน Loop จะทำให้โค้ดทำงานช้ามากเมื่อมีแถวจำนวนมาก  
   \*   \*\*คำแนะนำ:\*\* หากในอนาคตมีข้อมูลเยอะ ให้เก็บ Range ของแถวที่ต้องการไฮไลต์ไว้ใน Array แล้วค่อยสั่ง \`setBackground\` ทีเดียว (Batch Style) จะเร็วกว่ามากครับ  
3\. \*\*การป้องกันชื่อว่าง:\*\* ใน \`buildRecommendText\_\` คุณทำได้ดีมากครับที่ตรวจสอบ \`personResult.personId\` ก่อนแสดงผล แนะนำให้เพิ่มการตรวจสอบ \`\!isCompany\` หรือข้อมูลอื่นๆ ในทำนองเดียวกัน เพื่อไม่ให้ข้อความแนะนำดูผิดพลาดหากผลการ Normalize ล้มเหลว

\#\#\# สรุปภาพรวม  
โค้ดในชุดนี้ \*\*"ผ่านเกณฑ์การเขียนระดับโปรดักชั่น"\*\* ครับ มีการทำความสะอาดข้อมูล (Data Cleaning) หลังจากการตัดสินใจของมนุษย์ (Human-in-the-loop) ได้อย่างถูกต้องแม่นยำ

\#\#\# 🔧 การปรับแก้คำแนะนำ (Refinement)

\#\#\#\# 1\. เรื่อง \`srcObj.rawPlaceName\` ที่อาจไม่มีอยู่จริง  
ในคำแนะนำเดิมผมเสนอให้ใช้ \`srcObj.rawPlaceName\` แต่จากการตรวจสอบไฟล์ \`04\_SourceRepository.gs\` พบว่า \`buildSourceObj\_()\` \*\*ไม่ได้สร้างฟิลด์นี้ไว้\*\* ครับ

\*\*ข้อมูลจริงจาก \`04\_SourceRepository.gs\`:\*\*  
\`\`\`javascript  
// ใน buildSourceObj\_()  
rawPersonName: String(row\[SRC\_IDX.SHIP\_TO\_NAME\] || '').trim(), // ชื่อผู้รับ/สถานที่ (ผสมกัน)  
rawAddress:    String(row\[SRC\_IDX.SYS\_ADDRESS\] || '').trim(),  // ที่อยู่เต็ม  
// ไม่มี rawPlaceName แยกต่างหาก  
\`\`\`

\*\*✅ วิธีแก้ไขที่ถูกต้องสำหรับ \`enqueueReview\`:\*\*  
เนื่องจากชื่อสถานที่มักปนอยู่กับ \`SHIP\_TO\_NAME\` (ซึ่งกลายเป็น \`rawPersonName\` ใน srcObj) ควรใช้ค่านี้แทนครับ:

\`\`\`javascript  
// ✅ แก้ไขในฟังก์ชัน enqueueReview  
newRow\[REVIEW\_IDX.RAW\_PLACE\]     \= srcObj.rawPersonName;  // ใช้ชื่อจาก SHIP\_TO\_NAME แทน  
newRow\[REVIEW\_IDX.RAW\_SYS\_ADDR\]  \= srcObj.rawAddress;     // ที่อยู่เดิม  
\`\`\`

\> \*\*เหตุผล:\*\* ในระบบขนส่งไทย ชื่อผู้รับ (\`SHIP\_TO\_NAME\`) มักคือชื่อร้าน/บริษัท ซึ่งก็คือ "ชื่อสถานที่" นั่นเองครับ

\---

\#\#\#\# 2\. การแก้ไข \`applyReviewDecision\` ให้ดึงค่าจากชีตให้ถูกต้อง  
ในกรณี \`CREATE\_NEW\` ต้องดึงค่า "ที่อยู่" จากคอลัมน์ที่เก็บไว้ใน \`Q\_REVIEW\` จริงๆ ไม่ใช่ใช้ค่าชื่อสถานที่ซ้ำ

\*\*✅ โค้ดที่แนะนำ:\*\*  
\`\`\`javascript  
// ✅ แก้ไขในฟังก์ชัน applyReviewDecision (กรณี CREATE\_NEW)

// ดึงค่าจากชีตให้ถูกต้องตามคอลัมน์  
const rawPerson \= String(rowData\[REVIEW\_IDX.RAW\_PERSON\] || '').trim();  
const rawPlace  \= String(rowData\[REVIEW\_IDX.RAW\_PLACE\] || '').trim();   // ชื่อสถานที่  
const rawAddr   \= String(rowData\[REVIEW\_IDX.RAW\_SYS\_ADDR\] || '').trim(); // ที่อยู่จริง

// สร้าง srcObj ให้ถูกต้อง  
const srcObj \= {  
  rawPersonName: rawPerson,  
  rawAddress:    rawAddr,      // ✅ ใช้ที่อยู่จริง ไม่ใช่ชื่อสถานที่  
  rawLat:        Number(rowData\[REVIEW\_IDX.RAW\_LAT\]),  
  rawLng:        Number(rowData\[REVIEW\_IDX.RAW\_LNG\]),  
  province:      '',  
  deliveryDate:  new Date(),  
  hasGeo:        \!\!(Number(rowData\[REVIEW\_IDX.RAW\_LAT\])),  
};

// เรียกฟังก์ชันให้ถูกพารามิเตอร์  
const personResult \= resolvePerson(rawPerson);  
const placeResult  \= resolvePlace(rawPlace, rawAddr);  // ✅ ส่งชื่อสถานที่ และ ที่อยู่ แยกกัน  
const geoResult    \= resolveGeo(srcObj.rawLat, srcObj.rawLng);  
\`\`\`

# **aistudio**

จากการรีวิวโค้ดใน 12\_ReviewService.gs  
ในไฟล์นี้เป็นจุดตัดสินใจสุดท้ายที่รับค่าส่งต่อจากมนุษย์  
ระบบถือว่ามีคอนเซปต์ชัดเจนครับ อย่างไรก็ตามยังแฝงช่องโหว่และ Bug อยู่ 5 จุดสำคัญ  
ที่สามารถส่งผลร้ายแรงกับการสร้างประวัติส่งงานใน Database  
ข้ามตัวหรือทำงานไม่ผ่านได้  
ดังนี้ครับ:

🚨 1\. ข้อมูลสำคัญของใบเสร็จ "สูญหาย" ระหว่างทำ 'CREATE\_NEW'

ใน Section 2 ของฟังก์ชัน applyReviewDecision() ตรงที่ case 'CREATE\_NEW':  
คุณสร้างตัวแปรชั่วคราว srcObj มาหุ้มเองด้วยมือ  
แต่ดันตกหล่นฟิลด์สำคัญที่สุดของการยืนยันตัวตน  
นั่นก็คือ invoiceNo กับ sourceRow:

    case 'CREATE\_NEW':  
      // สร้างใหม่ทุกอย่าง  
      const srcObj \= {  
        rawPersonName: rawPersonName,  
        rawAddress:    rawPlaceName,  
        rawLat:        Number(rowData\[REVIEW\_IDX.RAW\_LAT\]),  
        rawLng:        Number(rowData\[REVIEW\_IDX.RAW\_LNG\]),  
        province:      '',  
        deliveryDate:  new Date(),  
        hasGeo:        \!\!(Number(rowData\[REVIEW\_IDX.RAW\_LAT\])),  
        // ❌ ตรงนี้ตัวที่จำเป็นขาดหายไป\!   
      };

ทำไมถึงร้ายแรง: พอสคริปต์มันเดินหน้าเรียก executeDecision(...)  
แล้วส่งไปจนถึงท้ายปลายน้ำในโมดูลที่ชื่อว่า  
upsertFactDelivery มันจะขาดกุญแจหลักอ้างอิงรอยต่อตาราง (sourceRow หรือ tx\_id  
ที่ผูกกับต้นทาง) สุดท้ายการ Create\_New ระบบจะพยายามบันทึกสถิติ Fact  
เปล่าๆ ไม่รู้ว่าไปเชื่อมข้อมูลตั๋วไหน ขยะในระบบก็จะทับซ้อนและ Crash  
ฝั่ง Engine ต้นทางด้วย

🛠 ทางแก้: นำของเหล่านี้อ้างอิงให้กลับเข้ามาเป็น Object เช่น:

        invoiceNo:  String(rowData\[REVIEW\_IDX.INVOICE\_NO\] || ''),  
        sourceRow:  Number(rowData\[REVIEW\_IDX.SOURCE\_ROW\] || 0),

🚨 2\. บั๊กเช็คตรรกะใน Bulk Apply — Comment เขียนแบบนึง โค้ดกลับเขียนเช็คอีกแบบ\!

ใน applyAllPendingDecisions ช่วงรวบคำสั่งตรวจที่เสร็จสิ้นจาก Sheet:

    // 💡 ดูสิ่งที่คุณ Note เอาไว้ "ประมวลผลเฉพาะ: มี Decision \+ Status ยังไม่ Done"  
    // ❌ แต่กลับเขียน If ตกไปว่า  
    if (decision && status \=== 'In\_Review') { 

จุดผิดปกติ: ตอนคุณเขียนลง Q\_REVIEW แรกเริ่ม สเตตัสถูกตั้งว่า "Pending"  
ซึ่งเวลาการใช้จริงแอดมินหรือมนุษย์ใน Sheets  
จะมาพิมพ์คอลัมน์เลือก "Decision"  
เลยและมักจะไม่ตั้งคอลัมน์ Status  
ทีละบรรทัดมานั่งเปลี่ยนเป็น In\_Review  
ทุกครั้ง ดังนั้นถ้า Status ในเซลล์มันโชว์ค้างคำว่า "Pending"  
เมื่อคุณมากดปุ่ม Bulk รัน  
โค้ดตรงนี้จะทำตัวแปลกแยกด้วยการมองข้ามทิ้งทั้งหมดเป็น  
"skipped"

🛠 ทางแก้: ปรับตรรกะให้เป็นไปตามคอมเมนต์ข้างต้นคือ:

    if (decision && status \!== 'Done') { 

🚨 3\. เสี่ยงตกม้าตายเรื่อง "Exception: Column Start at 1" จาก Case Sensitive

ตรงจุดท้ายฟังก์ชันของ applyReviewDecision ก่อนจะปรับสถานะ 'Done':

  const statusCol     \= headers.indexOf('status')      \+ 1;  
  const reviewedAtCol \= headers.indexOf('reviewed\_at') \+ 1;  
  sheet.getRange(reviewRow, statusCol).setValue('Done');

คุณมีความเสี่ยงเอา Headers มาค้น Index ด้วยข้อความเป๊ะๆ ซึ่งตาราง Google Sheets  
แถวแรกถ้ามีการใส่เป็นตัวอักษรใหญ่ผสมเช่น Status หรือเว้นวรรค  
คำสั่งนี้จะดีดตัวแปลตีผลกลับมาติดลบเป็น (-1)  
ทันที บวกเข้ากับ 1 ผลลัพธ์ตกที่ Index 0 ซึ่งการไป setValue ที่ Column เป็น 0  
แอปจะเกิด Error กระตุกหยุดสคริปต์ตาย

🛠 ทางแก้: ในฐานที่คุณมี Enum ค่า Constants ของฟิลด์เก็บไว้อยู่ตลอด  
ควรดึงแบบแข็งกระด้างอย่างเดียวปลอดภัย 100% :

  const statusCol     \= REVIEW\_IDX.STATUS \+ 1;  
  const reviewedAtCol \= REVIEW\_IDX.REVIEWED\_AT \+ 1;

⚠️ 4\. หลีกเลี่ยงตัวแปรหลงฝูงที่ล้นขอบ Scope (ES6 Syntax Hazard) ใน switch(decision)

ถ้าพิจารณา Case สร้างสโคปในโหมด JavaScript/AppScript โหมด V8 แล้ว ในส่วนนี้ครับ  
:

    case 'CREATE\_NEW':  
      // ประกาศในกรณี...  
      const srcObj \= { ...  
      const personResult \= resolvePerson(rawPersonName); 

หากภายในสคริปต์ Block เดียวของคำสั่ง switch(..) {}  โดยปกติการนิยามใช้ let และ  
const ไม่ถูกครอบสับอยู่ในตัวกรอบขอบปีกกาของตัวเอง  
เมื่อวันไหนต้องการให้โหมดอื่นนำคำใกล้ตัวคล้าย  
ๆ กันมารวม Case มันจะระเบิดออกทับตัวแปรทันที (Uncaught SyntaxError: Identifier  
'xxx' has already been declared)

🛠 ทางแก้ (เชิงโครงสร้างไวยากรณ์ Code Best Practice): ครอบกรณีตัวแปรเยอะที่พ่นลง  
case เดี่ยวๆ ให้แน่นด้วยวงเล็บปีกกาสร้างบล็อกสเปซไว้

    case 'CREATE\_NEW': {    // \<\< เปิด scope  
       const srcObj \= {...}  
       ...   
       break;  
    }  // \<\< ปิดปีกกา
