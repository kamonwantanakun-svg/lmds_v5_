# **perplexity**

ไฟล์ \`17\_SearchService.gs\` เป็นโมดูลที่มีความสำคัญสูงมากและเขียนออกมาได้ \*\*"ครบเครื่อง (Comprehensive)"\*\* ในเชิงตรรกะการตัดสินใจ (Decision Flow) ครับ การทำ Tiered Lookup (A ถึง E) ช่วยให้ระบบมีความยืดหยุ่นสูงมากในการจัดการข้อมูลที่อาจจะไม่สมบูรณ์

จากการตรวจสอบโค้ดอย่างละเอียด ไม่พบจุดที่ผิดพลาดในเชิง Syntax แต่มีประเด็นที่อยากให้ตรวจสอบเพื่อให้ระบบรันได้อย่างราบรื่นที่สุด ดังนี้ครับ:

\#\#\# ข้อสังเกตเชิงเทคนิค

| จุดที่พบ | ปัญหา | วิธีแก้ไข / ข้อเสนอแนะ |  
| :--- | :--- | :--- |  
| \*\*ความผิดพลาดของ \`dests\` ใน Tier A\*\* | ใน Tier A: \`const dests \= getDestsByPersonAndPlace(personId, placeId);\` และตรวจสอบ \`dests.length \=== 1\` หากในชีตข้อมูลมีแถวที่เป็น \`ARCHIVED\` แต่ยังไม่ได้ถูกกรองออกในฟังก์ชัน \`getDestsByPersonAndPlace\` จะทำให้ \`dests.length\` เกินความจริง | ในฟังก์ชัน \`getDestsByPersonAndPlace\` คุณได้ใส่เงื่อนไข \`d.status \!== APP\_CONST.STATUS\_ARCHIVED\` ไว้แล้ว ถือว่า \*\*ถูกต้องและรอบคอบมากครับ\*\* |  
| \*\*การใช้ \`bgColorArr\` และ \`setBackground\`\*\* | คุณใช้ \`sheet.getRange(i \+ 2, 1, 1, SCHEMA.DAILY\_JOB.length).setBackground(colorRow\[0\])\` ภายใน Loop ของการ \`forEach\` ซึ่งการเรียก API ของ Google Sheets ทุกครั้งใน Loop จะทำให้การประมวลผล \*\*ช้าลงมาก\*\* (Latency สูง) | หาก \`totalRows\` มีจำนวนมาก แนะนำให้ใช้ \`Range.setBackgrounds(colors)\` (รับเป็น 2D Array) เพื่อทำ Batch Set พื้นหลังทีเดียว จะเร็วขึ้นแบบเห็นได้ชัดครับ |  
| \*\*การใช้ \`row\[DATA\_IDX.SHIP\_TO\_ADDR\]\`\*\* | ใน \`runLookupEnrichment\` คุณใช้ \`rawPlace\` จาก \`DATA\_IDX.SHIP\_TO\_ADDR\` แต่ใน \`findBestGeoByPersonPlace\` คุณใช้ค่าเดียวกันนี้เป็นทั้ง \`rawAddress\` และ \`rawAddress\` (ส่งเข้าไป 2 ครั้ง) | โค้ดไม่มีปัญหาครับ แต่อาจลองพิจารณาแยก \`rawAddress\` และ \`province\` (ถ้ามี) ส่งเข้าไปให้ \`resolvePlace\` เพื่อความแม่นยำขึ้น |

\#\#\# ข้อแนะนำเพิ่มเติมเพื่อความปลอดภัย (Pro Tips)  
1\. \*\*Time Guard:\*\* ใน \`runLookupEnrichment\` ไม่มีการทำ Time Guard (ป้องกัน Timeout 6 นาที) เหมือนใน \`MatchEngine\` หากชีต \`DAILY\_JOB\` มีข้อมูลหลักหลายพันแถว อาจเกิดอาการ \`Exceeded maximum execution time\` ได้ แนะนำให้นำ Logic \`Time Guard\` มาปรับใช้ครับ  
2\. \*\*การทำ Batch Write:\*\* การใช้ \`writeRange.setValues(latActualArr)\` เป็นวิธีที่ถูกต้องและดีที่สุดสำหรับการเขียนข้อมูลจำนวนมากแล้วครับ (ทำได้ดีเยี่ยม)  
3\. \*\*การประมวลผลแถวเดิม:\*\* คุณมีการเช็ก \`if (existingLL && existingLL.includes(','))\` เพื่อข้ามแถวที่มีพิกัดแล้ว ซึ่งเป็นวิธีที่ดีมากสำหรับการทำระบบที่ต้องรันซ้ำบ่อยๆ (Idempotency)

\#\#\# สรุปภาพรวม  
โค้ดในชุดนี้ \*\*"ผ่านเกณฑ์การเขียนระดับโปรดักชัน"\*\* โดยเฉพาะการจัดการสถานะ \`NOT\_FOUND\` และ \`Fallback\` ต่างๆ ที่ออกแบบมาอย่างรัดกุม

\*\*สรุป:\*\* คุณสามารถรวมไฟล์นี้เข้าสู่โปรเจกต์ได้เลยครับ ระบบทำงานได้ตาม Logic ที่วางไว้ และที่สำคัญคือมีการจัดการเรื่อง Batch Operations (Batch Read/Write) มาอย่างดีเยี่ยมครับ\!

# **qwen**

จากการตรวจสอบโค้ด \`17\_SearchService.gs\` อย่างละเอียด \*\*พบจุดที่เขียนผิดทางตรรกะ (Logic Error) 1 จุด\*\* ที่อาจทำให้ระบบค้นหาสถานที่ไม่แม่นยำครับ

\#\#\# 🚨 1\. จุดที่เขียนผิด: ส่งพารามิเตอร์ซ้ำใน \`resolvePlace\`  
ในฟังก์ชัน \`findBestGeoByPersonPlace()\` มีการเรียกใช้ \`resolvePlace()\` โดยส่งค่า \`rawPlace\` เข้าไปทั้งสองพารามิเตอร์:

\`\`\`javascript  
// บรรทัดที่ \~55  
const placeResult \= resolvePlace(rawPlace, rawPlace);  
//                                  ❌ ผิด: พารามิเตอร์ที่ 1 ควรเป็น "ชื่อสถานที่"  
\`\`\`

\*\*ผลกระทบ:\*\*  
ฟังก์ชัน \`resolvePlace(name, address)\` จากไฟล์ \`07\_PlaceService.gs\` expects:  
1\.  พารามิเตอร์ที่ 1: \*\*ชื่อสถานที่\*\* (เช่น "ไทวัสดุ", "เซเว่น")  
2\.  พารามิเตอร์ที่ 2: \*\*ที่อยู่\*\* (เช่น "123 ถ.รัชดา ลาดยาว")

การส่ง \`rawPlace\` (ซึ่งจากข้อมูล \`DAILY\_JOB\` คือ \`ShipToAddress\` หรือ "ที่อยู่") เข้าไปทั้งสองช่อง จะทำให้ระบบขาดข้อมูล "ชื่อสถานที่" ในการตัดคำหรือจับคู่ ส่งผลให้การค้นหาสถานที่ล้มเหลวหรือได้คะแนนความแม่นยำต่ำครับ

\*\*วิธีแก้ไข:\*\*  
เนื่องจากในข้อมูลขนส่งไทย ชื่อผู้รับ (\`ShipToName\`) มักคือชื่อร้านหรือบริษัท ควรส่ง \`rawPerson\` เป็นพารามิเตอร์แรกแทนครับ:

\`\`\`javascript  
// ✅ ถูกต้อง: ส่งชื่อสถานที่ (จาก rawPerson) และ ที่อยู่ (จาก rawPlace)  
const placeResult \= resolvePlace(rawPerson, rawPlace);  
\`\`\`

\> \*\*หมายเหตุ:\*\* หากกรณีไหนชื่อผู้รับเป็น "ชื่อบุคคล" ไม่ใช่ชื่อร้าน ระบบ \`resolvePlace\` มีตรรกะจัดการอยู่แล้ว แต่การส่งข้อมูลชื่อเข้าไปย่อมดีกว่าการส่งที่อยู่ซ้ำครับ

\#\#\# ✅ 2\. สิ่งที่เขียนถูกต้องแล้ว (สังเกตว่าทำได้ดีมาก)  
\*   \*\*ตรรกะการค้นหาแบบ Tier:\*\* การเรียงลำดับความสำคัญจาก \`Person+Place\` → \`Place Only\` → \`Person Only\` → \`SCG API Fallback\` เป็นกลยุทธ์ที่ \*\*ฉลาดและสมเหตุสมผล\*\* มากสำหรับการค้นหาพิกัดที่มีความไม่แน่นอนสูง ✅
