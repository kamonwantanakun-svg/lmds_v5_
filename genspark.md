| คอลัมน์ | ชื่อ | คำอธิบาย |  
|---------|------|----------|  
| A | \`รหัสไปรษณีย์\` | 5 หลัก |

\#\#\# MODULE 2: \`MOD\_GEO.gs\` — ระบบจัดการพิกัด & ภูมิศาสตร์

\*\*จุดประสงค์:\*\* จัดการทุกเรื่องที่เกี่ยวกับ LatLong, reverse geocode, และ SYS\_TH\_GEO

| ฟังก์ชัน | หน้าที่ |  
|---------|---------|  
| \`parseLatLong(rawStr)\` | แยก \`"13.123,100.456"\` → \`{lat, lng}\` |  
| \`isValidLatLong(lat, lng)\` | ตรวจสอบว่าอยู่ในพื้นที่ไทยหรือไม่ |  
| \`calcDistanceMeters(lat1, lng1, lat2, lng2)\` | คำนวณระยะทาง 2 จุด (Haversine) |  
| \`isSameGeoPoint(lat1, lng1, lat2, lng2, radiusM)\` | ตรวจว่าพิกัด 2 จุดอยู่ใน radius เดียวกันหรือไม่ |  
| \`lookupTHGeo(lat, lng)\` | ค้นหาข้อมูล SYS\_TH\_GEO จากพิกัด → คืน \`{tambon, amphoe, province, postcode, note}\` |  
| \`lookupTHGeoByText(text)\` | ค้นหา SYS\_TH\_GEO จากข้อความที่อยู่ |  
| \`reverseGeocode(lat, lng)\` | เรียก Google Maps API เพื่อแปลงพิกัด → ที่อยู่ข้อความ |  
| \`clusterGeoPoints(rows, radiusM)\` | จัดกลุ่มแถวที่มีพิกัดใกล้กัน |  
| \`loadTHGeoCache()\` | โหลด SYS\_TH\_GEO ลง cache (memory) เพื่อความเร็ว |  
| \`matchAddressToTHGeo(addressText)\` | เทียบข้อความที่อยู่กับ SYS\_TH\_GEO แบบ fuzzy |  
| \`extractProvinceFromText(text)\` | ดึงชื่อจังหวัดออกจากข้อความ |  
| \`extractAmphoeFromText(text)\` | ดึงชื่ออำเภอออกจากข้อความ |  
| \`initTHGeoSheet()\` | สร้างชีต SYS\_TH\_GEO \+ header ถ้ายังไม่มี |

\#\#\# MODULE 3: \`MOD\_NORMALIZE.gs\` — ระบบทำความสะอาดข้อความ

\*\*จุดประสงค์:\*\* normalize ชื่อบุคคล และชื่อสถานที่ให้เป็นมาตรฐาน

| ฟังก์ชัน | หน้าที่ |  
|---------|---------|  
| \`normalizeName(rawName)\` | ลบคำนำหน้า (คุณ/นาย/นาง/ช่าง/ร้าน/พี่ ฯลฯ), trim, lower |  
| \`normalizeAddress(rawAddress)\` | ตัดคำฟุ่มเฟือย, ทำให้สม่ำเสมอ |  
| \`removePrefixTitle(name)\` | ลบคำนำหน้าชื่อคน |  
| \`removeSuffixNoise(name)\` | ลบ suffix ที่ไม่ใช่ชื่อ เช่น "รับของ", "สาขา" |  
| \`extractPersonName(rawText)\` | พยายามดึงชื่อบุคคลออกจากข้อความปน |  
| \`extractBranchInfo(text)\` | ดึงข้อมูลสาขาออกจากชื่อสถานที่ |  
| \`tokenizeThai(text)\` | ตัดคำภาษาไทยแบบเบสิก |  
| \`calcFuzzyScore(str1, str2)\` | คำนวณความคล้ายของ 2 ข้อความ (Levenshtein \+ Thai-aware) |  
| \`calcNameSimilarity(name1, name2)\` | คะแนนความคล้ายชื่อบุคคล (น้ำหนักพิเศษสำหรับชื่อไทย) |  
| \`calcAddressSimilarity(addr1, addr2)\` | คะแนนความคล้ายที่อยู่ |  
| \`stripAllNoise(text)\` | ลบทุกอย่างที่เป็น noise ออก |

\#\#\#\# \`calcFuzzyScore(str1, str2)\` — ต้องออกแบบพิเศษสำหรับภาษาไทย

\`\`\`  
อัลกอริทึมที่ใช้รวมกัน:

1\. Exact Match          → score \= 100  
2\. Normalized Match     → score \= 95  (หลัง normalize แล้วเหมือนกัน)  
3\. Contains Match       → score \= 80  (str1 อยู่ใน str2 หรือกลับกัน)  
4\. Levenshtein Distance → score ตามความห่าง  
5\. Token Sort Ratio     → เรียงคำใหม่แล้วเปรียบ (สมชาย วงษ์ \= วงษ์ สมชาย)  
6\. Token Set Ratio      → เปรียบเฉพาะคำที่มีร่วมกัน

สุดท้าย → weighted average ของทั้งหมด  
\`\`\`

\#\#\#\# \`calcNameSimilarity(name1, name2)\` — น้ำหนักพิเศษไทย

\`\`\`  
น้ำหนักที่ใช้:  
\- ชื่อแรกตรงกัน (first token)    : \+40 คะแนน  
\- ชื่อหลังตรงกัน (last token)   : \+30 คะแนน    
\- fuzzy score รวม                : \+30 คะแนน

กรณีพิเศษ:  
\- ชื่อสั้นมาก (\< 3 ตัว) → penalty \-20  
\- มีตัวเลขปน → penalty \-10  
\- เป็น common word เช่น "ช่าง" → penalty \-30  
\`\`\`

\#\# 🎯 การออกแบบ Scoring System — ใช้กับปัญหาทั้ง 8 ข้อ

\#\#\# แผนภาพการตัดสินใจ

\`\`\`  
แถวใหม่เข้ามา  
     ↓  
\[STEP 1\] ตรวจคุณภาพข้อมูล  
     ├── ชื่อคน: detectNameQuality()    → score\_name\_quality  
     ├── ที่อยู่: detectAddressQuality() → score\_addr\_quality  
     └── พิกัด:  isValidLatLong()       → geo\_valid (true/false)  
     ↓  
\[STEP 2\] ค้นหา Candidates  
     ├── ค้นจากชื่อ:  findPersonCandidates()  → candidates\_by\_name\[\]  
     ├── ค้นจากพิกัด: findPersonByGeo()        → candidates\_by\_geo\[\]  
     └── รวมและ deduplicate candidates  
     ↓  
\[STEP 3\] Scoring แต่ละ Candidate  
     สำหรับแต่ละ candidate:  
     ├── name\_score     \= calcNameSimilarity(rawName, candidate.name)  
     ├── geo\_score      \= geoDistance \< 50m ? 100 : (\< 200m ? 70 : 0\)  
     ├── addr\_score     \= calcAddressSimilarity(rawAddr, candidate.addr)  
     ├── alias\_bonus    \= มีใน alias table ? \+20 : 0  
     └── TOTAL\_SCORE    \= (name\_score × 0.40)   
                        \+ (geo\_score  × 0.40)  
                        \+ (addr\_score × 0.15)  
                        \+ (alias\_bonus× 0.05)  
     ↓  
\[STEP 4\] ตัดสินใจ  
     ├── TOTAL ≥ 90  → AUTO CONFIRM → เขียน Master ทันที  
     ├── TOTAL 60–89 → REVIEW QUEUE → รอ Admin  
     ├── TOTAL \< 60  → CREATE NEW  → สร้าง record ใหม่  
     └── ไม่มี candidates → CREATE NEW  
\`\`\`

\# นิยามข้อมูลดิบที่ถูกต้องของโปรเจกต์นี้

ด้านล่างนี้คือความหมายที่ผมเข้าใจจากคำอธิบายล่าสุดของคุณ และผมจะยึดความหมายนี้เป็นหลักต่อจากนี้

\---

\#\# 1\) คอลัมน์ \`จุดส่งสินค้าปลายทาง\`  
\#\#\# ความหมาย  
เป็น \*\*ตัวเลขพิกัด LatLong ของจุดส่งจริง\*\*    
อยู่ในรูปข้อความหรือค่าที่เก็บเป็นคู่พิกัด เช่น

\`\`\`text  
13.123456,100.123456  
\`\`\`

และเป็นพิกัดแบบ \*\*ทศนิยม 6 หลัก\*\*

\#\#\# สถานะความน่าเชื่อถือ  
ถือว่าเป็น \*\*ข้อมูลสำคัญมาก\*\* และเป็นหนึ่งในคอลัมน์ที่น่าเชื่อถือที่สุด    
เพราะเป็นจุดที่คนขับกดส่งงานจริง

\#\#\# บทบาทในระบบ  
คอลัมน์นี้ควรถูกมองเป็น:

\- \*\*Geo Point หลักของงานส่ง\*\*  
\- ต้นทางของการสร้าง \`LAT\`  
\- ต้นทางของการสร้าง \`LONG\`  
\- ต้นทางของการแปลงเป็นที่อยู่ข้อความใน \`ชื่อที่อยู่จาก\_LatLong\`  
\- ใช้เป็นตัวช่วยสำคัญในการแก้ปัญหาเรื่องข้อมูลมั่วของ \`ชื่อปลายทาง\` และ \`ที่อยู่ปลายทาง\`

\#\#\# สรุปสั้น  
คอลัมน์นี้คือ \*\*“พิกัดจริงของงานส่ง”\*\*

\---

\#\# 2\) คอลัมน์ \`ชื่อเจ้าของสินค้า\`  
\#\#\# ความหมาย  
คือ \*\*ชื่อบริษัทที่ขายของ\*\*    
ไม่ใช่ชื่อลูกค้าปลายทาง    
ไม่ใช่ชื่อสถานที่ส่ง    
ไม่ใช่ชื่อคนขับ

\#\#\# ตัวอย่างแนวคิด  
เช่น  
\- SCG  
\- บริษัท A  
\- บริษัท B

บริษัทพวกนี้คือ “เจ้าของสินค้า” หรือ “ผู้ขาย”

\#\#\# สถานะความน่าเชื่อถือ  
โดยทั่วไปถือว่า \*\*ค่อนข้างน่าเชื่อถือ\*\* ในฐานะข้อมูลเชิงธุรกิจ    
แต่ \*\*ไม่ควรใช้เป็นตัวระบุปลายทางโดยตรง\*\*

\#\#\# บทบาทในระบบ  
คอลัมน์นี้ควรใช้เพื่อ:

\- แยกบริบทธุรกิจว่า ลูกค้ารายเดียวกันอาจซื้อของจากหลายบริษัท  
\- ช่วยวิเคราะห์ alias ของลูกค้าบางกรณี  
\- ช่วยทำรายงาน เช่น ลูกค้าคนนี้รับสินค้าจากบริษัทไหนบ้าง  
\- ใช้เป็น metadata ของธุรกรรมส่งสินค้า

\#\#\# สิ่งที่ไม่ควรทำ  
ไม่ควรเอาคอลัมน์นี้ไปใช้แทน:  
\- \`ชื่อปลายทาง\`  
\- \`ที่อยู่ปลายทาง\`  
\- \`สถานที่ส่ง\`

\#\#\# สรุปสั้น  
คอลัมน์นี้คือ \*\*“บริษัทผู้ขาย/เจ้าของสินค้า”\*\*

\---

\#\# 3\) คอลัมน์ \`ชื่อปลายทาง\`  
\#\#\# ความหมาย  
คือ \*\*ชื่อบุคคลลูกค้าที่สั่งของ\*\*

และนี่คือคอลัมน์ที่คุณย้ำชัดว่าเป็น \*\*ตัวปัญหาหลัก\*\*

\#\#\# ปัญหาของคอลัมน์นี้  
ข้อมูลในคอลัมน์นี้อาจมีลักษณะดังนี้

\- เขียนไม่ครบ  
\- เขียนไม่เหมือนกัน  
\- มีคำนำหน้า/ไม่มีคำนำหน้า  
\- มีชื่อเล่น  
\- มีคำแทรก เช่น ร้าน, ช่าง, พี่, คุณ  
\- บางครั้งเป็นชื่อคน  
\- บางครั้งปนชื่อร้าน  
\- บางครั้งตัดคำมามั่ว  
\- บางครั้งคนเดียวกันแต่คนละบริษัทเรียกไม่เหมือนกัน

\#\#\# ตัวอย่างปัญหา  
เช่น คนเดียวกันอาจถูกเขียนเป็น

\- สมชาย  
\- คุณสมชาย  
\- ช่างสมชาย  
\- สมชาย รับของ  
\- สมชาย ร้านวัสดุ  
\- นายสมชาย

ทั้งหมดนี้อาจเป็นคนเดียวกัน หรืออาจไม่ใช่ก็ได้

\#\#\# สถานะความน่าเชื่อถือ  
เป็นข้อมูลที่ \*\*สำคัญมาก แต่สกปรกมาก\*\*  
จึง \*\*ห้ามใช้เป็น primary key\*\*  
และ \*\*ห้ามใช้ตัดสินปลายทางเพียงคอลัมน์เดียว\*\*

\#\#\# บทบาทในระบบ  
คอลัมน์นี้ต้องถูกใช้เป็น:

\- \`raw\_person\_name\`  
\- วัตถุดิบสำหรับสร้าง \`Person Master\`  
\- วัตถุดิบสำหรับ \`Person Alias\`  
\- หนึ่งในตัวช่วยแก้ปัญหา 8 ข้อ

\#\#\# สรุปสั้น  
คอลัมน์นี้คือ \*\*“ชื่อบุคคลลูกค้า”\*\*    
แต่เป็นข้อมูลดิบที่ \*\*ต้อง normalize \+ match \+ review\*\*

\---

\#\# 4\) คอลัมน์ \`LAT\`  
\#\#\# ความหมาย  
คือ \*\*ตัวเลขละติจูดแบบทศนิยม 6 หลัก\*\*    
และถูกแปลงมาจากคอลัมน์ \`จุดส่งสินค้าปลายทาง\`

\#\#\# ความสัมพันธ์กับคอลัมน์อื่น  
\`LAT\` และ \`LONG\` คือผลจากการแยกค่าพิกัดออกมาจาก

\- \`จุดส่งสินค้าปลายทาง\`

เช่น

\- \`จุดส่งสินค้าปลายทาง\` \= \`13.123456,100.123456\`  
\- \`LAT\` \= \`13.123456\`  
\- \`LONG\` \= \`100.123456\`

\#\#\# สถานะความน่าเชื่อถือ  
ถือว่า \*\*น่าเชื่อถือสูง\*\*  
เพราะมาจากพิกัดจริงชุดเดียวกัน

\#\#\# บทบาทในระบบ  
ใช้สำหรับ:

\- สร้าง \`M\_GEO\_POINT\`  
\- เทียบจุดซ้ำ / จุดใกล้เคียง  
\- reverse geocode  
\- ค้นหาปลายทางจากพิกัด  
\- ช่วยแยกว่า “ชื่อเหมือนกันแต่คนละที่”  
\- ช่วยแยกว่า “คนละชื่อแต่พิกัดเดียวกัน”

\#\#\# สรุปสั้น  
คอลัมน์นี้คือ \*\*ละติจูดจริงของจุดส่ง\*\*

\---

\#\# 5\) คอลัมน์ \`LONG\`  
\#\#\# ความหมาย  
คือ \*\*ตัวเลขลองจิจูดแบบทศนิยม 6 หลัก\*\*    
และถูกแปลงมาจากคอลัมน์ \`จุดส่งสินค้าปลายทาง\`

\#\#\# สถานะความน่าเชื่อถือ  
เช่นเดียวกับ \`LAT\` คือ \*\*น่าเชื่อถือสูง\*\*

\#\#\# บทบาทในระบบ  
ใช้ร่วมกับ \`LAT\` เสมอ เพื่อเป็น:

\- พิกัดจริงของปลายทาง  
\- ตัวระบุ geo point  
\- ตัวช่วยในการจับคู่ destination  
\- ตัวช่วยตรวจความถูกต้องของชื่อ/ที่อยู่

\#\#\# สรุปสั้น  
คอลัมน์นี้คือ \*\*ลองจิจูดจริงของจุดส่ง\*\*

\---

\#\# 6\) คอลัมน์ \`ที่อยู่ปลายทาง\`  
\#\#\# ความหมาย  
คือ \*\*ชื่อสถานที่อยู่\*\* ที่มาจากระบบเดิม    
และคุณย้ำว่าอันนี้ก็เป็น \*\*ตัวปัญหาอีกตัวหลัก\*\*

\#\#\# ลักษณะของข้อมูล  
ข้อมูลนี้อาจเป็น

\- ชื่อที่อยู่  
\- ชื่อสถานที่  
\- ชื่อร้าน  
\- ข้อความประกอบที่อยู่  
\- ข้อความตัดมาไม่ครบ  
\- ข้อความมั่วจากระบบ  
\- ข้อความไม่มาตรฐาน

\#\#\# ประเด็นสำคัญ  
คอลัมน์นี้ \*\*ไม่ใช่พิกัดจริง\*\*  
และ \*\*ไม่ใช่ที่อยู่ที่เชื่อถือได้ 100%\*\*  
แต่เป็น “ข้อความที่ระบบเดิมพยายามให้เป็นที่อยู่”

\#\#\# สถานะความน่าเชื่อถือ  
อยู่ระดับ \*\*กลางถึงต่ำ\*\*  
เพราะเป็นข้อความมั่วจากระบบ

\#\#\# บทบาทในระบบ  
ควรใช้เป็น:

\- \`raw\_place\_address\`  
\- วัตถุดิบสำหรับสร้าง \`Place Master\`  
\- ใช้เทียบกับ \`ชื่อที่อยู่จาก\_LatLong\`  
\- ใช้เทียบกับ \`SYS\_TH\_GEO\`  
\- ใช้ช่วย scoring แต่ไม่ควรเป็นตัวตัดสินหลัก

\#\#\# สิ่งที่ไม่ควรทำ  
ไม่ควรใช้คอลัมน์นี้:  
\- เป็น key หลักของสถานที่  
\- merge สถานที่โดยตรงทันที  
\- สรุปว่า address เดียวกันแน่ ๆ จากข้อความเพียงอย่างเดียว

\#\#\# สรุปสั้น  
คอลัมน์นี้คือ \*\*“ข้อความที่อยู่/สถานที่จากระบบเดิม”\*\*    
มีประโยชน์ แต่ \*\*สกปรกและต้องตรวจซ้ำ\*\*

\#\# 7\) คอลัมน์ \`ระยะทางจากคลัง\_Km\`  
\#\#\# ความหมาย  
คือ \*\*ตัวเลขระยะทาง\*\* ที่คำนวณจาก

\- ต้นทาง \= คลัง SCG วังน้อย  
\- ปลายทาง \= \`จุดส่งสินค้าปลายทาง\`

และถูกสร้างโดยสคริปต์

\#\#\# สถานะความน่าเชื่อถือ  
ถือว่า \*\*มีประโยชน์เชิงวิเคราะห์\*\*  
แต่ไม่ใช่ข้อมูลหลักสำหรับระบุตัวตนลูกค้า

\#\#\# บทบาทในระบบ  
ใช้เพื่อ:

\- วิเคราะห์ว่าปลายทางนี้อยู่ไกลจากคลังเท่าไร  
\- ใช้ตรวจความผิดปกติ  
\- ใช้ทำรายงาน  
\- ใช้เช็ค sanity ของข้อมูล เช่น ถ้าพิกัดผิดมาก ระยะทางอาจผิดธรรมชาติ

\#\#\# สิ่งที่ไม่ควรทำ  
ไม่ควรใช้เป็นตัว match หลักของ customer/place    
เพราะระยะทางเดียวกันไม่ได้แปลว่าเป็นที่เดียวกัน

\#\#\# สรุปสั้น  
คอลัมน์นี้คือ \*\*“ระยะทางเชิงคำนวณจากคลังไปจุดส่ง”\*\*

\---

\#\# 8\) คอลัมน์ \`ชื่อที่อยู่จาก\_LatLong\`  
\#\#\# ความหมาย  
คือ \*\*ชื่อสถานที่อยู่แบบข้อความ\*\*    
ที่ถูกแปลงมาจากคอลัมน์ \`จุดส่งสินค้าปลายทาง\` โดยใช้ script

พูดให้ตรงคือ มันเป็นผลของ \*\*reverse geocoding\*\*

\#\#\# ความแตกต่างจาก \`ที่อยู่ปลายทาง\`  
สองคอลัมน์นี้คนชอบสับสนมาก จึงต้องแยกให้ชัด:

\#\#\#\# \`ที่อยู่ปลายทาง\`  
\- มาจากระบบเดิม  
\- เป็นข้อความที่ระบบต้นทางให้มา  
\- มั่วได้มาก

\#\#\#\# \`ชื่อที่อยู่จาก\_LatLong\`  
\- มาจากพิกัดจริง  
\- ระบบแปลงจาก LatLong เป็นข้อความ  
\- โดยทั่วไปน่าเชื่อถือกว่าในเชิงตำแหน่ง  
\- แต่บางครั้ง wording อาจไม่ตรงภาษาคนใช้งานจริง

\#\#\# สถานะความน่าเชื่อถือ  
โดยทั่วไปถือว่า \*\*น่าเชื่อถือมากกว่า \`ที่อยู่ปลายทาง\`\*\*  
ในฐานะ “ข้อความที่สะท้อนตำแหน่งจริง”

\#\#\# บทบาทในระบบ  
คอลัมน์นี้สำคัญมาก เพราะควรใช้เป็น:

\- \`geo\_resolved\_address\`  
\- ตัวช่วย normalize สถานที่  
\- ตัวช่วยเทียบกับ \`SYS\_TH\_GEO\`  
\- ตัวช่วยเช็คอำเภอ/ตำบล/จังหวัด  
\- ตัวช่วยแยกสาขา  
\- ตัวช่วยตรวจว่าที่อยู่จากระบบเดิมเพี้ยนแค่ไหน

\#\#\# สรุปสั้น  
คอลัมน์นี้คือ \*\*“ที่อยู่ข้อความที่สร้างจากพิกัดจริง”\*\*

\# สรุปความสัมพันธ์ของ 8 คอลัมน์นี้แบบสั้นและชัด

\#\# กลุ่มพิกัดจริง  
คอลัมน์ที่น่าเชื่อถือสูงสุดในเชิงตำแหน่งคือ

\- \`จุดส่งสินค้าปลายทาง\`  
\- \`LAT\`  
\- \`LONG\`

คอลัมน์พวกนี้สะท้อน \*\*ตำแหน่งจริงของจุดส่ง\*\*

\---

\#\# กลุ่มชื่อบุคคล  
คอลัมน์ที่บอกว่า “ใครเป็นลูกค้า” คือ

\- \`ชื่อปลายทาง\`

แต่มันเป็นข้อมูลดิบที่สกปรกมาก ต้องทำ alias/matching/review

\---

\#\# กลุ่มชื่อสถานที่/ที่อยู่  
คอลัมน์ที่เกี่ยวกับที่อยู่มี 2 ตัว

\- \`ที่อยู่ปลายทาง\` \= ข้อความมั่วจากระบบ  
\- \`ชื่อที่อยู่จาก\_LatLong\` \= ข้อความที่แปลงจากพิกัดจริง

ถ้าให้เรียงความน่าเชื่อถือในเชิงตำแหน่ง:

1\. \`LAT/LONG\`  
2\. \`ชื่อที่อยู่จาก\_LatLong\`  
3\. \`ที่อยู่ปลายทาง\`

\---

\#\# กลุ่มข้อมูลบริบทธุรกิจ  
คอลัมน์ที่ช่วยบอกบริบทธุรกรรมคือ

\- \`ชื่อเจ้าของสินค้า\`  
\- \`ระยะทางจากคลัง\_Km\`

สองตัวนี้มีประโยชน์มาก แต่ไม่ใช่ key หลักของการระบุตัวตนปลายทาง

\# สิ่งที่การตีความใหม่นี้ทำให้ต้อง “แก้โค้ดเชิงตรรกะ”

จากคำอธิบายล่าสุดของคุณ มีจุดสำคัญที่ต้องย้ำให้ระบบคิดใหม่ให้ถูก:

\#\# 1\) \`ชื่อปลายทาง\` \= คน ไม่ใช่สถานที่  
ดังนั้นในโค้ด matching:  
\- ต้องส่งเข้า person logic เป็นหลัก  
\- ห้ามเอาไปใช้แทน place โดยตรงแบบเหมารวม

\#\# 2\) \`ที่อยู่ปลายทาง\` \= ข้อความที่อยู่จากระบบเดิมที่มั่ว  
ดังนั้น:  
\- ใช้เป็น input ของ place/address scoring ได้  
\- แต่ห้ามเชื่อ 100%

\#\# 3\) \`ชื่อที่อยู่จาก\_LatLong\` \= address ที่ derive จากพิกัดจริง  
ดังนั้น:  
\- ควรมีน้ำหนักมากกว่า \`ที่อยู่ปลายทาง\`  
\- ใช้เทียบกับ \`SYS\_TH\_GEO\` ได้ดีมาก

\#\# 4\) \`LAT/LONG\` มาจาก \`จุดส่งสินค้าปลายทาง\`  
ดังนั้น:  
\- ถ้ามี \`LAT/LONG\` แล้ว ถือเป็นค่าหลัก  
\- \`จุดส่งสินค้าปลายทาง\` เป็น raw source สำรอง/ตรวจย้อนกลับ

\#\# 5\) \`ชื่อเจ้าของสินค้า\` ไม่ใช่ผู้รับ  
ดังนั้น:  
\- อย่าเอาไปสร้าง person master ของลูกค้า  
\- แต่สามารถใช้เป็น context ว่า “ลูกค้าคนเดียวกันอาจซื้อจากหลายเจ้าของสินค้า”

\# ถ้าจะเขียนเป็นคู่มือให้ทีมใช้งาน ควรอธิบายแบบนี้

\#\# คำอธิบายสั้นสำหรับทีม  
\- \`ชื่อปลายทาง\` \= ชื่อลูกค้าบุคคล  
\- \`ที่อยู่ปลายทาง\` \= ที่อยู่จากระบบเดิม อาจมั่ว  
\- \`ชื่อที่อยู่จาก\_LatLong\` \= ที่อยู่จากพิกัดจริง  
\- \`LAT/LONG\` \= พิกัดจริงของจุดส่ง  
\- \`ชื่อเจ้าของสินค้า\` \= บริษัทผู้ขาย  
\- \`ระยะทางจากคลัง\_Km\` \= ระยะจากคลังวังน้อยถึงจุดส่ง

\# ข้อสรุปสำคัญที่สุดสำหรับโปรเจกต์นี้

ถ้าจะพูดให้สั้นและตรงที่สุด:

\- \*\*ตัวจริงของตำแหน่ง\*\* คือ \`จุดส่งสินค้าปลายทาง / LAT / LONG\`  
\- \*\*ตัวจริงของคน\*\* คือ \`ชื่อปลายทาง\` แต่ข้อมูลมั่ว ต้อง clean  
\- \*\*ตัวช่วยเรื่องที่อยู่\*\* คือ \`ชื่อที่อยู่จาก\_LatLong\`  
\- \*\*ตัวที่ต้องระวังมาก\*\* คือ \`ที่อยู่ปลายทาง\`  
\- \*\*ตัวบริบทธุรกิจ\*\* คือ \`ชื่อเจ้าของสินค้า\`  
\- \*\*ตัววิเคราะห์เสริม\*\* คือ \`ระยะทางจากคลัง\_Km\`
